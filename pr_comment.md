Thanks for the PR! The WebSocket auth handshake architecture looks great, and the delayed connection resolving in the frontend is nicely handled. However, there are a few issues to address before this can be merged:

1. **Linux/macOS Build Failure (`commands.rs`):** The Rust code uses `libc::getuid()` on Unix platforms, but the `libc` crate is missing from `tauri-app/src-tauri/Cargo.toml`. Because this was only tested on Windows, it will currently break the build on Unix systems. Please add `libc = "0.2"` to the `[target.'cfg(unix)'.dependencies]` section in `Cargo.toml`.
2. **Browser Dev Mode Breakage (`daemon.ts`):** In `daemon.ts`, if Tauri's `invoke` fails (such as when running the UI locally in standard browser dev mode), the auth token defaults to `""`. Since the Python daemon now strictly requires a valid token, local UI development will be completely disconnected. Consider adding a fallback for browser development, like reading from an environment variable (e.g., `import.meta.env.VITE_DAEMON_TOKEN`).
3. **Type Safety on Token (`server.py`):** A client sending `{"token": null}` in their JSON will result in `provided_token = None`, and `str(None)` will evaluate to the string `"None"`. While it won't match the securely generated token, it is cleaner to explicitly type-check with `isinstance(provided_token, str)` to prevent strange behavior.
